# 가챠패스 (상시패스)

> **요약:** 누적 가챠 횟수에 연동되는 패스 보상으로 뽑기 구매의 목표를 강화하고 천장 역할을 겸하는 상시 운영 시스템
> **현재 상태:** 미적용

> **목적:** 누적 가챠 횟수에 연동되는 패스 보상으로, 뽑기 구매의 목표를 강화하고 천장 역할을 겸하여 구매 경험의 편차를 줄인다
> **핵심 제한사항:** 기간 제한 없는 상시 운영, 사이클 완주 후 자동으로 다음 사이클 연결, CSV 데이터 드리븐 설계
> **비전 연결:** "습관 형성 → 장기 성장" 단계에서 가챠 행위 자체에 보상 루프를 부여하여, 뽑기를 단발성 도박이 아닌 목표 지향 활동으로 전환
> **상시 운영 근거:** 이 시스템은 천장(pity system) 역할을 겸한다. 천장이 기간에 따라 소멸하면 유저의 누적 진행도가 사라지고 신뢰가 깨지기 때문에, 일반 기간 한정 이벤트와 달리 상시 운영을 원칙으로 한다. "뽑을수록 반드시 손해 없다"는 보장이 이 시스템의 핵심 가치이며, 이것이 기간 한정 운영 원칙의 예외 근거다.

## 2. 달성 목표

### 플레이어 경험 목표
- 뽑기를 할 때마다 패스가 진행되어, "꽝이어도 손해 보지 않는다"는 안전감 제공
- 패스 초반·중간·최종에 핵심 보상을 배치하여 "몇 회만 더 뽑으면 확정 보상"이라는 명확한 목표 제시
- 사이클 완주 후 다음 사이클이 자동 연결되어, 뽑기를 계속할수록 가치가 쌓이는 성장 경험
- 프리미엄 미구매자도 무료 트랙으로 기본 보상 획득 → 과금 비과금 모두 혜택

### BM 목표
- **가챠 구매 촉진**: "패스 8스텝까지 3회만 더 뽑으면 되는데" → 추가 뽑기 유도
- **프리미엄 패스 판매**: 9,900원 단일 상품으로 뽑기 행위의 가성비 극대화
- **천장 역할**: 사이클 최종 보상이 등급 확정 선택권 → 확률 불운에 대한 보상 보장
- **장기 과금**: 사이클 반복 + 보상 상향으로 고래 유저의 지속 목표 제공

### 운영/개발 목표
- 기존 `PassListData` / `PassDetailData` 구조를 재활용하여 최소 개발 스코프
- `pass_group` 확장 + `unlock_condition_type` 확장으로 CSV 드리븐 운영
- **상시패스** 카테고리 아래 가챠패스 외에도 캐시패스, 레벨패스 등 동일 구조로 확장 가능
- 뽑기 종류별(장비/유닛) 독립 패스로 운영하여 각각의 뽑기 콘텐츠 활성화

## 3. 수행 방향

### 핵심 설계 원칙
1. **뽑기 = 패스 진행**: 뽑기 행위 자체가 패스의 진행 조건. 별도 미션이나 퀘스트 없이 자연스럽게 연동
2. **CSV 드리븐 사이클**: 사이클당 스텝 수, 필요 뽑기 횟수, 보상 모두 테이블에서 관리. 코드 변경 없이 밸런스 조정 가능
3. **상시패스 프레임워크**: 가챠패스를 첫 사례로, 향후 캐시구매패스·레벨패스·스테이지패스 등을 동일 구조로 확장할 수 있는 범용 프레임워크

### 제약 사항 / 금지 사항
- 패스 진행도는 리셋되지 않음 (상시 운영, 사이클 내 진행도 영구 보존)
- 프리미엄 패스 구매는 사이클 도중에도 가능하며, 이미 도달한 스텝의 프리미엄 보상을 소급 수령 가능
- 서버 권한 판정 필수 — 뽑기 횟수 카운트와 보상 지급은 서버에서 처리
- 패스 보상으로 뽑기 티켓을 지급할 경우, 해당 뽑기가 다시 패스를 진행시킴 (순환 허용, 단 밸런스 주의)

### 우선순위
| 우선순위 | 항목 | 이유 |
|---------|------|------|
| P0 (필수) | 가챠 횟수 → 패스 스텝 진행 핵심 루프 | 시스템의 존재 이유 |
| P0 (필수) | 무료/프리미엄 트랙 분리 | 기존 패스 구조 재활용, 과금 전환의 핵심 |
| P0 (필수) | 사이클 완주 후 자동 연결 | 장기 운영의 핵심 차별점 |
| P1 (중요) | 프리미엄 패스 소급 수령 | 구매 시점 제약 해소 → 매출 증가 |
| P1 (중요) | 사이클별 보상 상향 | 고래 유저 장기 목표 |
| P2 (선택) | 스텝 스킵 (캐시로 즉시 진행) | 추가 매출 창구, 기존 step_skip_fee 활용 |

## 4. 개요

### 기본 구성

```
[상시패스] (ContentsUnlock으로 해금)
 ├── [장비 가챠패스] (ContentsUnlock으로 해금, 장비 뽑기 연동)
 ├── [유닛 가챠패스] (ContentsUnlock으로 해금, 유닛 뽑기 연동)
 ├── [캐시 구매패스] (향후 확장)
 └── [레벨패스]     (향후 확장)
```

- **패스 유형**: 장비 가챠패스, 유닛 가챠패스 (각각 독립 운영)
- **진행 조건**: 해당 뽑기의 누적 횟수 (1회 뽑기 = 1카운트, 10연차 = 10카운트)
- **트랙**: 무료 트랙 + 프리미엄 트랙 (기존 Free / Special 구조)
- **사이클**: 최종 스텝 도달 후 다음 사이클 자동 시작 (무한 반복)
- **운영 방식**: 상시 (기간 제한 없음)

### 핵심 규칙
- 뽑기 1회 = 패스 진행 1카운트 (10연차 = 10카운트)
- 광고 무료 뽑기도 카운트에 포함
- 스텝별 필요 카운트는 테이블에서 지정 (예: 3회/스텝)
- 보상은 스텝 도달 시 수동 수령 (자동 수령 아님)
- 미수령 보상은 사이클 연결 시에도 소멸하지 않음

### 진입 조건 / 해금 조건
- **상시패스 카테고리**: `ContentsUnlockData`로 해금 (예: 튜토리얼 완료 또는 특정 스테이지 클리어)
- **장비 가챠패스**: 장비 뽑기 콘텐츠 해금 시 자동 해금
- **유닛 가챠패스**: 유닛 뽑기 콘텐츠 해금 시 자동 해금

## 5. 시스템 동작 흐름

### 전체 흐름 다이어그램
```
[뽑기 실행] → [서버: 뽑기 결과 + 패스 카운트 증가]
     → [패스 스텝 도달 확인] → [UI 알림 (레드닷)]
          → [유저: 보상 수령 버튼] → [서버: 보상 지급]
               → [최종 스텝 완료?] → [YES: 다음 사이클 자동 시작]
                                    → [NO: 계속 진행]
```

### 단계별 상세 흐름

#### Step 1. 진입
- 상시패스 UI 진입 경로: 로비 메인 HUD → 상시패스 버튼 (또는 뽑기 화면 내 패스 바로가기)
- ContentsUnlockData 조건 미달성 시 버튼 비표시 또는 잠금 표시

#### Step 2. 뽑기 → 카운트 증가
- 유저가 해당 뽑기(장비/유닛)를 실행
- 서버에서 뽑기 처리 시 해당 가챠패스의 누적 카운트를 함께 증가
- 1회 뽑기 = +1, 10연차 = +10
- 광고 무료 뽑기도 동일하게 카운트

#### Step 3. 스텝 도달 확인
- 증가된 카운트가 현재 스텝의 필요 카운트를 충족하는지 서버에서 판정
- 복수 스텝 동시 달성 가능 (10연차로 3스텝 건너뛰는 경우)

#### Step 4. 보상 수령
- 수령 가능 스텝에 레드닷 표시
- 유저가 수동으로 보상 수령 버튼 클릭
- 무료 트랙: 모든 유저 수령 가능
- 프리미엄 트랙: 프리미엄 패스 구매자만 수령 가능
- 프리미엄 미구매 상태에서 스텝 도달 → 이후 구매 시 소급 수령

#### Step 5. 사이클 완료 → 다음 사이클 연결
- 최종 스텝(10스텝) 보상 수령 완료 시
- 서버에서 사이클 카운터 +1, 스텝 0으로 리셋
- 누적 뽑기 카운트는 현재 사이클 기준으로 재시작 (또는 잔여분 이월)
- 다음 사이클의 보상 테이블 적용 (상향 보상 시 별도 테이블 참조)

### 예외 처리 / 엣지 케이스
| 상황 | 처리 방식 |
|------|----------|
| 10연차로 복수 스텝 동시 달성 | 모든 달성 스텝의 보상을 개별 수령 가능 |
| 프리미엄 패스 구매 전 이미 진행 | 구매 시점에 도달한 모든 프리미엄 보상 소급 수령 |
| 사이클 완주 후 보상 미수령 | 미수령 보상은 소멸하지 않음. 다음 사이클 진행과 병행 가능 |
| 보상으로 받은 뽑기 티켓 사용 | 해당 뽑기의 패스 카운트에 정상 반영 (순환 허용) |
| 네트워크 실패 | 서버 응답 대기 → 재시도. 뽑기 결과와 패스 진행은 원자적 처리 |
| 뽑기 상품 추가/삭제 | 새로운 뽑기 상품은 기존 패스에 연동하거나 신규 패스 생성 (테이블 설정) |

## 6. 데이터시트 구조

### 테이블 목록
| 테이블명 | 역할 | 비고 |
|---------|------|------|
| `PassListData` | 패스 시즌/유형 정의 | 기존 테이블 확장 (`pass_group` 추가값) |
| `PassDetailData` | 패스 스텝별 보상 정의 | 기존 구조 그대로 활용 |
| `ShopPassData` | 프리미엄 패스 상품 | 기존 구조 활용 |
| `ShopDrawData` | 뽑기 상품 정의 | 기존 데이터 참조 (연동 대상) |
| `ContentsUnlockData` | 해금 조건 | 상시패스 카테고리 + 개별 패스 해금 |

### 기존 테이블 현황

#### `PassListData` (현재 5행, 시즌패스 전용)
| 필드명 | 타입 | 설명 | 현재 값 예시 |
|--------|------|------|------------|
| id | int | 패스 ID | 10001~10005 |
| pass_group | string | 패스 카테고리 | "Season" (현재 유일) |
| buff_step | int | 버프 스텝 | 1~5 |
| step_skip_fee_type | string | 스텝 스킵 비용 타입 | "Currency" |
| step_skip_fee_id | int | 스텝 스킵 비용 ID | 2000000 |
| linked_product_id | int | 연결 상품 ID | 70001~70005 |

**확장 필요 사항:**
- `pass_group`에 `"Gacha"` 값 추가
- 연동 뽑기 상품 지정 필드 필요 (예: `linked_draw_id`)
- 사이클 관련 필드 필요 (예: `cycle_step_count`, `max_cycle`)

#### `PassDetailData` (현재 155행)
| 필드명 | 타입 | 설명 | 현재 값 예시 |
|--------|------|------|------------|
| id | int | 패스 ID | 10001 |
| step | int | 스텝 번호 | 1~31 |
| max_repeat_amount | int | 반복 상한 | 0 (현재 미사용) |
| unlock_condition_type | string | 진행 조건 | "Energy_Use" |

### 사이클별 보상 상향 기준

#### 채택 방식: 옵션 A — 사이클별 독립 PassDetailData

사이클마다 별도의 PassDetailData 행 세트를 등록한다. 예: 사이클 1용 pass_id=20001, 사이클 2용 pass_id=20011, 사이클 3+용 pass_id=20021 식으로 구성.

**장점:** 각 사이클의 보상을 완전히 자유롭게 설계할 수 있다. 기획자가 사이클별 보상 구성을 직접 테이블에서 확인하고 수정하기 쉽다.

**단점 (주의):** 사이클 수가 늘어날수록 PassDetailData 행이 선형으로 증가한다. 사이클 3개만 해도 30스텝 × 3 = 90행 추가. 사이클이 많아지면 테이블 관리 부담이 커지므로, 일정 사이클 이상은 단일 테이블로 고정하는 방식(예: 사이클 3+는 모두 동일 테이블)을 고려한다.

| 사이클 | pass_id | 최종 보상 (프리미엄) | 상향 방향 |
|--------|---------|---------------------|----------|
| 사이클 1 | 20001 / 20002 | Epic 확정 선택권 (장비) / Legendary 확정 선택권 (유닛) | 기본 |
| 사이클 2 | 20011 / 20012 | Epic 확정 + 강화 재료 번들 / Legendary 확정 + 파편 추가 | 부가 보상 증가 |
| 사이클 3+ | 20021 / 20022 (고정) | [TODO] 상위 등급 해금 or 특수 보상 | 추후 결정 |

> ⚠️ 세부 보상 구성은 테스트 후 확정합니다.

#### 대안: 옵션 B — 사이클 배율 컬럼

PassListData에 `cycle_reward_multiplier` 컬럼을 추가하고, 사이클 번호에 따라 보상 수량에 배율을 자동 적용하는 방식. 예: 사이클 1 = ×1.0, 사이클 2 = ×1.1, 사이클 3+ = ×1.2.

**장점:** 테이블 행 수가 사이클과 무관하게 고정된다. 대규모 확장에 유리.

**단점:** 배율이 소수점이면 보상 수량이 어색해질 수 있고, 사이클마다 질적으로 다른 보상(예: 사이클 2에서 새로운 보상 종류 추가)을 표현하기 어렵다. 코드 구현이 필요하다.

> 현재는 옵션 A로 진행하되, 운영 중 테이블 관리가 과도해지면 옵션 B 전환을 검토한다.

### 검증 기준
- **무과금 기준**: 일일 광고 뽑기 + 자연 획득 티켓으로 약 2~3주에 1사이클 완주 (무료 트랙 보상만)
- **소과금 기준** (프리미엄 9,900원): 프리미엄 보상 포함 시 뽑기 투자 대비 150%+ 가치 환수 체감
- **고과금 기준**: 다회 사이클 반복으로 보상 상향 혜택. 월 3~5사이클 완주 가능

| 프리미엄 구매 | 가챠패스 상세 → 구매 버튼 | 프리미엄 패스 IAP 구매 |

### 화면별 상세

#### 상시패스 메인
- **레이아웃:** 가로 탭 또는 세로 리스트로 패스 종류 나열
- **주요 요소:**
  - 각 패스의 현재 사이클 / 스텝 진행도 미리보기
  - 수령 가능 보상이 있으면 레드닷 표시
  - 잠금 상태인 패스는 해금 조건 표시

#### 가챠패스 상세
- **레이아웃:** 기존 시즌패스 UI (`SeasonPassPopUpUI`) 구조 재활용
  - 상단: 패스 이름 + 현재 사이클 번호 + 진행 게이지 (현재 뽑기 횟수 / 다음 스텝 필요 횟수)
  - 중앙: 가로 스크롤 스텝 목록 (무료 트랙 위, 프리미엄 트랙 아래)
  - 하단: 프리미엄 구매 버튼 (미구매 시) / 현재 사이클 정보
- **주요 요소:**
  - 핵심 보상 스텝 (is_main = TRUE)은 특별 연출/프레임으로 강조
  - 도달한 스텝: 수령 가능 상태 표시 (수령 완료 / 수령 대기)
  - 미도달 스텝: 잠금 + 남은 뽑기 횟수 표시
  - 사이클 완주 시: 축하 연출 → "다음 사이클 시작!" 안내
- **인터랙션:**
  - 보상 슬롯 탭 → 보상 상세 팝업 (아이콘, 수량, 설명)
  - 수령 가능 슬롯 탭 → 즉시 수령 + CommonRewardUI 연출
  - 전체 수령 버튼 → 수령 가능한 모든 보상 일괄 수령

#### 뽑기 화면 내 연동
- **주요 요소:**
  - 뽑기 화면 상단 또는 하단에 미니 패스 게이지 바 표시
  - "다음 보상까지 N회!" 텍스트로 뽑기 동기 부여
  - 게이지 바 탭 시 가챠패스 상세 화면으로 이동

### UX 주의사항
- 뽑기 결과 화면에서 패스 진행 상황을 간략히 표시 ("+3 진행! 다음 보상까지 2회")
- 사이클 완주 시 별도 축하 연출이 뽑기 결과 연출과 겹치지 않도록 순차 처리
- 프리미엄 미구매 유저에게 프리미엄 보상을 흐릿하게 표시하여 구매 욕구 자극
