## 2. 달성 목표

### 플레이어 경험 목표
- 계정 생성 직후부터 소액 구매 시 즉각적인 보상을 받는 경험 (개인화)
- 대규모 업데이트 때 추가 충전 보상으로 신규 콘텐츠 진입이 가속되는 경험 (서버)
- 두 이벤트가 동시에 열리면 같은 결제로 양쪽 보상을 모두 받는 경험

### BM 목표
- 개인화 이벤트: 신규 유저 초기 4주간 소액 과금 습관 형성 → 이후 월정액/상시 상품으로 자연 전환
- 서버 이벤트: 대규모 업데이트 시 전체 유저 대상 단기 매출 부스트
- 포인트 공유로 병렬 진행 시 과금 효율 극대화 체감

### 운영/개발 목표
- 두 이벤트 타입 모두 동일한 테이블 구조 공유 → 개발 1회, 운영 N회
- 보상/마일스톤은 CSV 데이터 드리븐 → 기획자가 운영 중 자유 조정
- 다국가 결제 대응을 위한 결제 포인트 정규화 구조

### 설계 의도

#### 왜 개인화 + 서버 2타입인가
- 상시 무한 반복은 보상이 "기본값"이 되어 체감 약화 → 과금 동기가 아닌 의무감
- 개인화(1회성 4주): 초기 습관 형성에 집중. 종료 후 자연스럽게 다른 과금 경로로 전환
- 서버(업데이트 시): 특별감과 FOMO 유지. 비활성 기간이 있어야 다음 이벤트의 가치가 살아남

#### 왜 4주(28일)인가
- 행동 심리학에서 습관 형성에 약 3~4주 소요
- 개인화: 4주면 일일 트랙 4회전 + 누적 마일스톤 1회 완주로 충분한 보상 체험
- 서버: 업데이트 흥행 주기와 일치. 4주 후 자연 종료로 다음 이벤트 기대감 유지

#### 왜 일일 충전은 주 5일 목표인가
- 7일 완주 목표는 주말 미접속 유저에게 "어차피 못 채운다"는 심리 유발
- 5일이면 평일만 구매해도 달성 가능 → 진입 장벽 최소화
- 주 7일 중 5일이므로 1~2일 빠져도 완주 가능 → 이탈 방지

#### 왜 tier별 탭을 분리하는가
- 본 게임의 핵심 타겟은 월 0~5만 원 수준의 소과금 유저
- 기준선을 높게 단일화하면 소과금 유저가 첫날부터 이탈, 낮게 단일화하면 고과금 유저에게 보상 가치가 없다
- tier 분리로 이 문제를 해결하면서, 중/고과금 탭을 자연스러운 업셀링 경로로 활용
- 각 탭이 독립 달성이므로 소과금 유저가 상위 탭을 보며 위축되지 않음

#### 일일 충전 보상 방향 (잠정)
- 소과금 유저의 핵심 동기는 "캐릭터가 강해지는 느낌"이므로 장비 아이템을 기본 보상으로 제안
- 구체적인 등급과 수치는 밸런스 파트에서 별도 확정

| 단계 | 보상 방향 |
|------|----------|
| 1~2일 | 장비 재료, 강화석 등 소모성 |
| 3~4일 | 저등급 장비 or 장비 선택권 |
| 5일 (완주) | 중등급 장비 or 전용 장비 |

## 3. 수행 방향

### 핵심 설계 원칙
1. **2타입 공통 구조**: 개인화/서버 이벤트 모두 동일한 테이블 구조(ChargeDailyRewardData, ChargeAccumulateData) 사용. EventManagerData의 `event_type`으로 구분
2. **데이터 드리븐 유연성**: 일일 충전 금액 단위(탭 수/기준 포인트), 보상 단계 수, 누적 마일스톤 구간을 CSV만으로 변경 가능
3. **다국가 정규화**: 결제 금액을 결제 포인트로 변환하여 국가별 통화 차이를 흡수. 모든 판정 기준은 포인트 단위
4. **"대표 상품 1개 = 당일 달성" 원칙**: 각 tier의 기준 금액은 해당 tier 대표 상품 1개 구매로 달성되도록 설정
5. **포인트 공유**: 동일 결제 1건의 charge_point가 활성 상태인 모든 충전 이벤트(개인화 + 서버)에 동시 반영

## 4. 시스템 구조

### 이벤트 타입 비교

| 항목 | 개인화 이벤트 | 서버 이벤트 |
|------|-------------|------------|
| 트리거 | 계정 생성 시 자동 활성화 | 운영팀이 수동 오픈 (대규모 업데이트 시) |
| 스코프 | 계정 단위 (Account) | 채널/서버 단위 (Channel) |
| 기간 | 4주 (28일) 고정 | 4주 (28일) |
| 횟수 | 계정당 1회 | 운영 판단에 따라 복수 |
| 일일 충전 | 7일 주기 × 4회전, 주 5일 목표 | 7일 주기 × 4회전, 주 5일 목표 |
| 누적 충전 | 4주간 마일스톤 (reset_type = none) | 4주간 마일스톤 (reset_type = none) |
| 포인트 | 공유 (동일 결제가 양쪽에 동시 반영) | 공유 |

### 구조 다이어그램

```
충전 이벤트 시스템
├── [개인화] 계정 생성 시 자동 (4주, 계정당 1회)
│   ├── 일일 충전 탭 (tier별 서브탭)
│   │   ├── [소과금] 7일 중 5일 달성
│   │   ├── [중과금] 7일 중 5일 달성
│   │   └── [고과금] 7일 중 5일 달성
│   └── 누적 충전 탭
│       └── 마일스톤 N단계 (4주 한정, reset_type = none)
│
├── [서버] 운영팀 오픈 (4주, 업데이트 연동)
│   ├── 일일 충전 탭 (tier별 서브탭)
│   │   └── (개인화와 동일 구조, 보상은 별도 event_id)
│   └── 누적 충전 탭
│       └── 마일스톤 N단계 (4주 한정, reset_type = none)
│
└── 포인트 공유: IAP 결제 1건 → 활성 이벤트 전체에 charge_point 동시 적립
```

- **일일 충전**: 주 단위로 리셋 (4주간 총 4회). 금액 단위별 탭으로 분리하여 소/중/고과금 유저 각각에 맞는 보상 트랙 제공. 각 탭은 병렬 진행 가능 (고과금 유저가 소과금 탭도 동시 달성). 주 7일 중 5일 달성이 목표
- **누적 충전**: 이벤트 기간(4주) 동안 누적. 마일스톤 구간별 보상 수령. `reset_type = none`으로 1회 완주 후 종료
- **충전 랭킹**: 보류. §3 랭킹 검토 의견 참조

### 핵심 규칙

#### 결제 포인트 (가칭)

> **명칭 미확정**: "결제 포인트"는 가칭이며, 유저 노출 명칭 및 내부 용어는 별도 확정 필요. 데이터 필드명은 `charge_point`로 통일한다.

- 모든 IAP 결제 시 해당 상품의 `charge_point` 값만큼 포인트 적립
- **포인트 공유**: 적립된 charge_point는 현재 활성 상태인 **모든 충전 이벤트**에 동시 반영
  - 개인화만 활성 → 개인화에만 적립
  - 서버만 활성 → 서버에만 적립
  - 둘 다 활성 → 양쪽 모두 적립 (동일 포인트, 이중 차감 아님)
- 일일/누적 모두 이 포인트를 기준으로 판정
- `ios_pid` / `aos_pid` 필드가 있는 **모든 테이블**에 `charge_point` 및 `krw` 필드를 추가하여 매핑 (상세: §6 데이터시트)
- 국가별 통화 차이는 포인트 매핑으로 흡수

#### 일일 충전 판정
- **일 기준**: 서버 시간(UTC+9) 00:00 ~ 23:59
- **금액 단위별 탭**: 각 탭(tier)마다 `daily_threshold` 값이 다름. 해당 일에 적립된 charge_point 합산이 해당 탭의 기준치 이상이면 달성
- **병렬 진행**: 하루에 50,000원 이상 충전 시 소과금/중과금/고과금 탭 모두 동시 달성
- **연속 불필요**: 월/수/금만 충전해도 3일 달성
- **주 5일 목표**: 7일 중 5일 달성 시 해당 주차 완주. 6~7일 달성분은 보너스 보상 없음 (누적 포인트에만 반영)
- **달성 후 추가 충전**: 일일 달성은 탭별 하루 1회 고정. 달성 이후 동일 일자 추가 충전분은 누적 충전 포인트에만 반영
- **주간 리셋**: 이벤트 시작일 기준 7일마다 해당 요일 00:00 서버 시간에 리셋 (4주간 총 4회)
- **리셋 시 스냅샷**: 달성 기록 및 수령 가능 여부는 해당 주차 기준 서버 스냅샷 처리. 미수령 보상은 우편 자동 발송 후 이력 초기화
- **이벤트별 독립**: 개인화와 서버 이벤트가 동시 활성 시, 각 이벤트의 일일 달성은 독립 추적 (같은 결제로 양쪽 모두 달성 가능)

#### 누적 충전 판정
- **누적 기준**: 해당 이벤트 시작 이후 전체 charge_point 합산
- **마일스톤 달성**: 누적 포인트 ≥ 해당 단계 `condition_point` 시 보상 수령 가능
- **1회 완주**: `reset_type = none`. 최고 단계 수령 후 "완료" 상태. 이벤트 기간 내 완주를 목표로 설계
- **이벤트별 독립 누적**: 개인화와 서버 이벤트의 누적 포인트는 각각 별도 추적. 같은 결제가 양쪽에 동시 적립되지만, 달성 상태는 독립

> **누적 목표 금액**: 현재 100만원은 임시 설정값. 밸런스 검토 후 확정 필요. [TODO]

  - **권장**: 이벤트 활성화 시점에 해당 기간 결제 이력을 집계하여 charge_point 일괄 적립
  - 이를 통해 상점 해금 직후부터 충전해도 이벤트 보상을 놓치지 않음

### 진입 조건 / 해금 조건

#### 개인화 이벤트
- `event_scope_type = Account` — 계정 생성 기준
- `start_day_offset` / `end_day_offset` — 계정 생성 N일 후 시작, 28일 후 종료
- 계정당 1회. 종료 후 재오픈 불가

#### 서버 이벤트
- `event_scope_type = Channel` — 채널 기준 운영
- `start_day_offset` / `end_day_offset` — 운영팀이 지정한 기간 (통상 28일)
- `unlock_condition_type = Stage_Clear` 또는 `None`
- 업데이트 시 EventManagerData 행 추가로 오픈

#### 노출 기준

| 상태 | 노출 방식 |
|------|----------|
| 해금 전 | 로비에 버튼/아이콘 **완전 비노출** |
| 해금 후 ~ 종료 전 | 로비 좌측 숏컷 아이콘 노출. 수령 가능 보상 있을 때 레드닷 표시 |
| 이벤트 종료 후 | 숏컷 제거. 미수령 보상은 우편 자동 발송 |
| 누적 충전 "완료" 시 | 탭 유지, 내부 그레이아웃 + "모든 보상을 수령했습니다" 문구 |
| 병렬 활성 시 | 이벤트 목록에 개인화/서버 각각 표시. 진입 시 어떤 이벤트인지 명확히 구분 |

### 유저 여정

#### 개인화 이벤트 — 신규 유저

##### D+0 — 첫 진입
해금 조건 충족 시 로비 좌측에 숏컷 아이콘과 레드닷이 생긴다. 처음 진입하면 일일 충전 탭이 기본으로 열린다. 소과금 유저는 "1,500원이면 오늘 달성 가능하다"는 판단을 즉시 할 수 있다. 누적 충전 탭에서 마일스톤 전체 로드맵을 확인하며, 지금 충전하면 일일과 누적이 동시에 쌓인다는 것을 인지한다.

##### D+1~6 — 습관 형성 (1주차)
매일 로그인 시 레드닷으로 "수령할 보상이 있다"는 것을 인지한다. 5일 달성이 목표이므로 1~2일 빠져도 부담 없이 완주 가능. 소과금 유저는 가장 저렴한 상품 1개로 매일 달성 가능하므로 부담이 낮다.

##### D+7 — 1주차 완주 및 리셋
이벤트 시작일 기준 7일째 00:00에 리셋 발생. 미수령 보상은 우편 자동 발송. 완주 유저는 5일차 보상으로 성취감을 얻고 2주차 도전 동기가 형성된다.

##### D+7~27 — 2~4주차 반복
같은 구조로 3회 더 반복. 매주 리셋마다 "이번 주도 5일 채우자"는 루틴이 자리잡는다. 동시에 누적 포인트가 쌓이며 마일스톤 보상을 하나씩 수령한다.

##### D+28 — 이벤트 종료
4주가 지나면 이벤트가 자동 종료된다. 미수령 보상은 우편 발송. 4주간 형성된 소액 구매 습관은 이후 월정액/일일 패키지 등 상시 상품으로 자연 전환된다.

#### 서버 이벤트 — 대규모 업데이트

##### 업데이트 당일
서버 이벤트 오픈 공지. 기존 유저에게 새로운 충전 이벤트가 열리며, 업데이트 콘텐츠를 빠르게 즐기기 위한 과금 동기가 생긴다. 개인화 이벤트가 동시에 활성 상태라면 같은 결제로 양쪽 보상을 모두 받는다.

##### 4주간 운영
개인화 이벤트와 동일한 일일(5일/주) + 누적 구조. 업데이트 테마에 맞는 보상(신규 유닛 소환권, 전용 장비 등)으로 차별화.

##### 종료
이벤트 종료 → 보상 수령 기회 소멸 → 다음 업데이트까지 비활성. 비활성 기간이 있어야 다음 서버 이벤트의 특별감이 유지된다.

### 보상 구조 요약

| 트랙 | 보상 방식 | 보상 슬롯 | 리셋 주기 | 기간 |
|------|----------|----------|----------|------|
| 일일 충전 (소과금) | 일수별 단계 보상 (5단계) | 3슬롯 × 5단계 | 이벤트 시작일 기준 7일 주기 | 4주 (4회전) |
| 일일 충전 (중과금) | 일수별 단계 보상 (5단계) | 3슬롯 × 5단계 | 이벤트 시작일 기준 7일 주기 | 4주 (4회전) |
| 일일 충전 (고과금) | 일수별 단계 보상 (5단계) | 3슬롯 × 5단계 | 이벤트 시작일 기준 7일 주기 | 4주 (4회전) |
| 누적 충전 | 마일스톤 구간 보상 (N단계) | 3슬롯 × N단계 | 리셋 없음 (reset_type = none) | 4주 통합 |

> **일일 충전 단계 수 원칙**: 주 5일 달성 목표에 맞춰 5단계를 기본으로 한다. 모든 tier의 단계 수는 동일하게 맞추는 것을 원칙으로 한다.

## 5. 시스템 동작 흐름

### 전체 흐름 다이어그램

```
[이벤트 활성화]
├── 개인화: 계정 생성 + 해금 조건 충족 시 자동
└── 서버: 운영팀 EventManagerData 행 추가로 오픈
     ↓
[이벤트 시작 전 결제분 소급 적립 (개인화 전용, 선택)]
     ↓
[유저 IAP 결제] → [서버: 결제 완료]
     ↓
[서버: charge_point 적립]
     ├── 활성 이벤트 전체에 동시 적립 (포인트 공유)
     ├── 이벤트별 일일 포인트 합산 갱신 (tier별 달성 판정)
     └── 이벤트별 누적 포인트 합산 갱신
     ↓
[클라이언트: 충전 이벤트 상태 갱신]
     ├── 이벤트별 일일 충전 tier별 달성 여부 확인 → 레드닷
     └── 이벤트별 누적 마일스톤 도달 여부 → 레드닷
     ↓
[유저 보상 수령] → [서버 검증] → [CommonRewardUI]
     ↓
[주간 리셋 (시작일 기준 7일 주기, 4회)]
     ├── 일일 충전: 전 tier 일수/수령 이력 초기화. 미수령분 우편 발송
     └── 누적 충전: 리셋하지 않음 (계속 누적)
     ↓
[D+28: 이벤트 종료]
     └── 미수령 전체 보상 우편 발송. 이벤트 비활성화
```

### 단계별 상세 흐름

#### Step 1. 이벤트 활성화
- **개인화**: 계정 생성 + `start_day_offset`일 경과 시 자동 시작. 계정당 1회
- **서버**: 운영팀이 EventManagerData에 행 추가. 채널 전체 적용
- 클라이언트: `UserDataManager.UpdateEventData()` → `UserEventDatas`에 캐싱
- [개인화 전용, 선택] 이벤트 시작 전 결제 이력을 집계하여 charge_point 소급 적립

#### Step 2. 결제 및 포인트 적립
- 유저가 IAP 상품 결제 → `WASClient.ShopBuySend()` 호출
- 서버: 해당 상품의 `charge_point`를 유저 데이터에 적립
  - **활성 이벤트 전체**에 동시 적립 (이벤트별 독립 필드)
  - 이벤트별 `daily_charge_point` (당일 합산) 갱신
  - 이벤트별 `total_charge_point` (전체 누적) 갱신
- 서버 응답에 충전 이벤트 상태 포함 → 클라이언트 갱신

#### Step 3. 일일 충전 달성 판정
- 이벤트별 독립 판정: charge_point는 1일 1회 합산 적립. 각 tier의 `daily_threshold`와 **독립적으로 비교**
- 각 tier별로 `daily_charge_point >= daily_threshold[tier]` 이면 당일 달성 (탭별 하루 1회)
- tier별 주간 내 달성 일수 카운트 +1
- 달성 일수에 해당하는 보상 단계가 열림 (최대 5단계)
- 달성 이후 동일 일자에 추가 충전된 포인트는 누적 충전 포인트에만 반영
- 병렬 이벤트: 개인화 + 서버 모두 활성이면, 같은 결제로 양쪽 이벤트의 일일 달성이 동시 처리

#### Step 4. 누적 충전 마일스톤 판정
- 이벤트별 독립 누적: `total_charge_point >= condition_point[step]` 이면 해당 단계 수령 가능
- `reset_type = none`: 최고 단계 수령 후 "완료" 상태

#### Step 5. 보상 수령
- 유저가 UI에서 수령 버튼 클릭 → 서버 API 호출 (이벤트 ID + 트랙 타입 + 단계 지정)
- 서버: 수령 가능 여부 검증 후 보상 지급
- 클라이언트: `CommonRewardUI` 팝업으로 보상 표시

#### Step 6. 주간 리셋 (서버 스케줄러)
- 이벤트 시작일 기준 7일 주기, 해당 요일 00:00 서버 시간 실행 (4주간 총 4회)
- 일일 충전: 전 tier 달성 일수 초기화, 미수령 보상 우편 발송
- 누적 충전: 초기화하지 않음

#### Step 7. 이벤트 종료 (D+28)
- `end_day_offset` 도달 시 이벤트 비활성화
- 미수령 보상 전체 우편 발송 (일일 + 누적)
- UI 숏컷 제거

### 예외 처리 / 엣지 케이스
| 상황 | 처리 방식 |
|------|----------|
| 결제 후 앱 종료 (포인트 갱신 미수신) | 다음 접속 시 서버에서 최신 상태 동기화 |
| 주간 리셋 시 미수령 보상 | 우편으로 자동 발송 |
| 이벤트 종료 시 미수령 보상 | 일일 + 누적 전체 우편 발송 |
| 누적 reset_type=none인데 최고 단계 도달 | "완료" 상태 표시, 추가 보상 없음 |
| 결제 환불 | 서버에서 환불 감지 시 적립 포인트 차감. 이미 수령한 보상은 별도 정책 (프로그래머 논의) |
| 이벤트 기간 중 상품 가격/charge_point 변경 | 이미 적립된 포인트는 소급 변경하지 않음 |
| 동시 여러 건 결제 | 각 건별로 순차 적립, 합산 후 판정 |
| 이벤트 시작 전 결제분 (개인화) | 소급 적립 적용 (§4 참조) |
| 고과금 유저가 소과금 탭 동시 달성 | 정상 동작. 각 tier 독립 판정, 보상 모두 수령 가능 |
| 개인화 + 서버 병렬 활성 시 결제 | 양쪽 이벤트에 동일 charge_point 동시 적립. 각 이벤트 달성 독립 판정 |
| 서버 이벤트 도중 개인화 이벤트 종료 | 서버 이벤트는 영향 없이 계속. 이후 결제는 서버 이벤트에만 적립 |

| ShopCurrencyData | ✅ 있음 | `charge_point` 추가 |
| ShopPassData | ✅ 있음 | `charge_point` 추가 |
| ShopMembershipData | ✅ 있음 (`_krw`) | `charge_point` 추가. `_krw` → `krw` 정규화 여부 확인 필요 |
| ShopBeginnerEventPackageData | ❌ 없음 | `charge_point` + `krw` 추가 |
| ShopGlobalRecordProductData | ❌ 없음 | `charge_point` + `krw` 추가 |
| ShopGlobalRecordPackageData | ❌ 없음 | `charge_point` + `krw` 추가 |

> **다국가 처리 방식**: `charge_point`는 상품별로 기획자가 직접 세팅한다. 국가별 가격이 다르더라도 동일 상품은 동일 포인트를 부여하므로 환율 변동 영향 없음.
>
> **대안**: `krw` 필드를 기준으로 서버에서 변환 공식(`charge_point = krw / 15` 등)을 적용하는 방법도 가능하다. `krw`와 `charge_point`를 모두 갖추면 **기획자 직접 세팅 / 서버 자동 변환** 두 방식 모두 대응 가능하여 활용도가 가장 높다.

## 7. 밸런스 계획

### 설계 기준
- 일일 충전 소과금 탭: 1,500원 상당 최소 구매만으로 달성 → 5일 달성 시 구매액 대비 2~3배 가치 보상
- 일일 충전 중/고과금 탭: 해당 금액 구간 유저에게 추가 보상으로 업셀링 유도
- 누적 충전: 소과금(~10,000원) 유저도 3단계까지 도달 가능하도록 구간 설계
- 누적 최종 목표: 100만원 (임시) — 밸런스 검토 필요 [TODO]
- 보상은 게임 내 핵심 성장 재화 위주 (골드, 다이아, 에너지, 강화 재료)
- 서버 이벤트 보상: 업데이트 테마에 맞는 보상 (신규 유닛 소환권, 전용 장비 등)으로 개인화와 차별화

### 보상 밸류 배분 전략

5일 구조 확정 (주 7일 중 5일 달성 목표). 배분 방식은 아래 옵션 중 선택하여 **별도 문서로 확정** 필요.

| 옵션 | 설명 | 효과 |
|------|------|------|
| 초반 집중 | 1~2일차에 높은 가치 배분 | 이탈 방지, 빠른 보상 체감. 후반 동기 약화 우려 |
| 후반 집중 | 4~5일차에 높은 가치 배분 | 5일 완주 동기 극대화. 초반 이탈 시 효과 없음 |
| 균등 배분 | 매일 비슷한 가치 | 안정적이나 임팩트 약함 |
| 점진적 상승 (권장) | 1일차 소 → 5일차 대, 완만한 상승 곡선 | 매일 "내일은 더 좋다" 체감. 완주 시 최대 보상 |

> 구체적 보상 아이템/수량은 별도 밸런스 문서에서 정의한다.

### 검증 기준
- 소과금 기준 (월 1~2만원): 소과금 탭 주간 3~4일 달성, 누적 3단계(~10,000원) 도달
- 중과금 기준 (월 5~10만원): 소과금 탭 매주 5일 + 중과금 탭 3~4일, 누적 5~6단계
- 고과금 기준 (월 20만원+): 전 tier 매주 5일, 누적 빠른 완료

## 8. UI / UX

### 화면 목록

|--------|----------|------|
| ChargeEventUI | 로비 → 이벤트 버튼 | 충전 이벤트 메인 (탭 구조) |

### 화면별 상세

#### ChargeEventUI (메인 화면)
- **병렬 이벤트 처리**:
  - 개인화 + 서버 이벤트 동시 활성 시, 이벤트 선택 상단 탭 또는 드롭다운으로 전환
  - 각 이벤트별로 독립된 일일/누적 탭 표시
  - 이벤트 명칭(개인화 / 서버 이벤트명)으로 구분
- **레이아웃:**
  - 상단: 이벤트 제목 + 남은 기간 카운트다운 (종료일까지)
  - 중앙: 선택된 탭 콘텐츠 영역
  - 하단: 메인 탭 바 [일일 충전] [누적 충전]

#### [탭 그룹] 일일 충전
- **서브탭**: 금액 단위별 (소과금 / 중과금 / 고과금)
  - 탭 명칭은 `local_tier_name`에서 가져옴
  - 각 서브탭의 상단에 해당 tier 기준 금액 표시
- **레이아웃 (각 서브탭 동일 구조):**
  - 상단: 주간 달성 현황, 리셋 시간, 충전 현황, 주간 무료 보상
  - 중단: 일수별 보상 목록
    - 각 행: [N일] + 보상 아이콘 × 3 + [수령] 버튼
    - 상태: 미오픈(Off) / 이동(Lock) / 수령 가능(On) / 수령 완료 / 만료(해당 일차 경과 후 미달성)
  - 보상 목록 슬롯 순서는 고정하되, 오늘에 해당하는 슬롯 포커싱
- **인터랙션:**
  - [충전하기] 버튼 → 상점으로 이동
  - [수령] 버튼 → 서버 API → CommonRewardUI
  - 잠금 상태 탭 → 필요 일수 안내 토스트
  - 서브탭 전환 시 해당 tier 달성 현황으로 갱신
- **카운트다운 표기:**
  - 1~3주차: "N일 후 리셋"
  - 4주차: "N일 후 이벤트 종료"
- **만료 처리:** 해당 일차가 경과하면 미달성 슬롯을 만료 상태로 전환. 수령 불가, 별도 시각 처리(회색 처리 등)

#### [탭] 누적 충전
- **레이아웃:**
  - **is_repeat = False 인 경우:**
    - 상단: 누적 다이아 + 이벤트 남은 기간
  - **is_repeat = True 인 경우:**
    - 상단: 누적 다이아 + 이벤트 남은 기간 + 완주 횟수 (1 이상부터 표시) + 현재 진행 회차 (기본=1)
  - 중단: 마일스톤 목록 (스크롤)
    - 각 행 (is_repeat=False): [N단계 / 필요 다이아] + 보상 아이콘 × 3 + [수령] 버튼
    - 각 행 (is_repeat=True, 2회차~): [N단계 / 필요 다이아(직전회차 최종 요구치 합산)] + 보상 아이콘 × 3 + [수령] 버튼
    - 상태: 잠금 / 수령 가능 / 수령 완료
  - 보상 획득한 슬롯은 최하단으로 내려준다
- **인터랙션:**
  - [충전하기] 버튼 → 상점으로 이동
  - [수령] 버튼 → 서버 API → CommonRewardUI

### 레드닷 규칙
| 조건 | 표시 위치 |
|------|----------|
| 일일 충전 수령 가능 보상 있음 (어느 tier든) | 로비 이벤트 버튼, 일일 충전 탭, 해당 tier 서브탭 |

### 유저 데이터 필드 (UserDataManager)

> 이벤트별 독립 필드. `event_id`로 구분하여 저장.

| 필드명 | 타입 | 설명 |
|--------|------|------|
| charge_event_id | int | 대상 이벤트 ID |
| charge_event_daily_point | int | 당일 결제 포인트 합산 |
| charge_event_daily_achieved_tiers | dict (tier→bool) | 당일 tier별 달성 여부 |
| charge_event_weekly_days | dict (tier→int) | 이번 주 tier별 달성 일수 |
| charge_event_daily_reward_flags | dict (tier→int 비트마스크) | tier별 일일 보상 수령 이력 |
| charge_event_total_point | int | 누적 결제 포인트 |
| charge_event_accum_reward_flags | int (비트마스크) | 누적 보상 수령 이력 (step별 1비트) |

> **병렬 이벤트**: 활성 이벤트가 2개면 위 필드 세트가 event_id별로 2세트 존재.

## 부록: 충전 랭킹 (보류)

> 이 섹션은 보류된 랭킹 시스템의 설계안을 보관한다. 진행 결정 시 본문으로 이동.

### 보류 사유

1. **이미 충분한 동기 구조**: 일일 충전 트랙(tier별) + 누적 마일스톤만으로 명확한 구매 동기가 있다. 랭킹은 시스템을 복잡하게 만들 수 있다.
2. **채널 유저 풀 규모 문제**: 채널 단위 랭킹은 과금 유저가 충분히 많아야 경쟁이 형성된다. 채널 인원이 적으면 1~3위가 고정되어 의미가 없다.
3. **개발 비용 대비 효과 불확실**: 실시간 랭킹은 서버 부하/구현 비용이 있고, 고정 랭킹은 효과가 의문.

### 대안

| 대안 | 설명 | 비용 |
|------|------|------|
| A. 1위 브로드캐스팅만 남기기 | 주간 리셋 시 최다 충전 유저를 채널 공지 1회 노출. 기존 브로드캐스팅 시스템 활용 (폴리싱 병행) | 낮음 |
| B. 누적 충전 명예 스티커 | 특정 누적 마일스톤 달성 시 프로필 카드 전용 스티커 해제 | 낮음 |

> 랭킹은 유저 풀과 과금 패턴 데이터가 쌓인 뒤, 별도 이벤트로 독립 설계하는 것이 더 효과적이다.
