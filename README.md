# Meeting Knowledge Base — 시스템 구조 설명

이 시스템의 "학습" 메커니즘은 LLM이 각 작업마다 wiki를 보강하는 누적 구조다.

---

## 파일별 역할

### 입력 (읽기 전용)
| 파일/경로 | 역할 |
|---|---|
| `raw/meetings/` `raw/gdd/` `raw/feedback/` `raw/publisher/` | 원본 사실의 출처. 절대 수정 안 함. |

### 위키 핵심 구조
| 파일 | 역할 |
|---|---|
| `wiki/index.md` | **라우터** — 전체 문서 목록 + 한줄 요약. Query 시 LLM이 가장 먼저 읽어서 어디를 더 볼지 결정. |
| `wiki/log.md` | **append-only 이력** — 어떤 작업이 언제 이루어졌는지 기록. 중복 작업 방지 + 맥락 추적. |
| `wiki/concepts/` | 주제별 합성 지식. Query가 새로운 연결을 발견하면 여기가 보강된다. |
| `wiki/decisions/` | 확정된 결정사항. 새 회의록에서 결정이 번복되면 이 파일이 업데이트된다. |
| `wiki/timeline/` | 시간순 맥락. 같은 주제가 어떻게 변화했는지 추적 가능. |
| `wiki/people/` | 담당자별 관여 이력. 누가 어떤 결정에 관여했는지 연결. |

---

## "학습" 루프

```
raw/ (사실)
  ↓ /ingest
wiki/ (합성) ← 점점 정교해지는 공간
  ↓ /query
답변 + wiki 업데이트 (Two Outputs)
  ↓ /lint
불일치·누락 수정
  ↓
다음 질문 시 더 풍부한 wiki를 참조
```

**핵심 원리**: `/query`는 답변만 하지 않고, 질문에 답하는 과정에서 나온 새로운 합성(연결, 패턴, 함의)을 `wiki/concepts/`나 `wiki/decisions/`에 직접 기록한다. 즉 질문 자체가 wiki를 보강하는 작업이 된다. `log.md`는 이 누적 이력을 보존해 미래 LLM 세션에서도 "무엇이 이미 분석됐는지" 파악 가능하게 한다.
