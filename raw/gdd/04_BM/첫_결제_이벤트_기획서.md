# 첫 결제 이벤트

> **요약:** 무과금에서 소과금으로의 전환율 향상을 위한 조건달성보상 이벤트. 범용 ConditionRewardEvent 시스템의 첫 번째 인스턴스로 CSV 데이터 드리븐 구조 설계. 서버 권한 판정 기반이며 미수령 보상은 우편 자동 발송.
> **현재 상태:** 미적용

## 1. 제목

| 항목 | 내용 |
|------|------|
| 시스템 명칭 | 조건 달성 보상 이벤트 (ConditionRewardEvent) |
| 한 줄 설명 | 이벤트 기간 내 특정 조건 달성 시 보상을 지급하는 범용 이벤트 시스템. 첫 번째 인스턴스로 "첫 결제 이벤트" 운영 |

## 2. 달성 목표

### 플레이어 경험 목표
- 첫 결제에 대한 즉각적인 보상으로 긍정적 결제 경험 제공
- 이벤트 기간 한정 보상으로 결제 동기 부여

### BM 목표
- 무과금 유저의 소과금 전환율 향상
- 첫 결제 허들을 낮추어 과금 유저 풀 확대

### 운영/개발 목표
- 범용 "조건 달성 보상 이벤트" 타입으로 설계하여 다양한 조건(첫 던전 클리어, 레벨 달성 등)에 재활용
- CSV 데이터 드리븐 구조로 기획자가 보상/조건/기간을 자유롭게 세팅
- 기존 EventManagerData 기반으로 최소한의 신규 개발로 구현

## 3. 수행 방향

### 핵심 설계 원칙
1. **범용 이벤트 타입**: `EventInfoType.ConditionReward`로 등록. 조건/보상은 CSV에서 정의하며, 조건 타입만 변경하면 다른 이벤트로 재활용 가능
2. **기존 시스템 최대 활용**: EventManagerData(기간 관리), ConditionType(조건 판정), CommonRewardUI(보상 표시), Mail(우편 발송) 등 기존 인프라 활용
3. **서버 권한 판정**: 조건 달성 여부와 보상 지급은 서버(WAS)에서 최종 판정. 클라이언트는 표시/요청만 담당

### 제약 사항 / 금지 사항
- 클라이언트 단독으로 보상 지급 불가 (서버 검증 필수)
- 이벤트 기간 외 보상 수령 불가 (단, 조건 달성 후 미수령분은 우편으로 자동 발송)
- 동일 이벤트 인스턴스에서 보상 중복 수령 불가

### 우선순위
| 우선순위 | 항목 | 이유 |
|---------|------|------|
| P0 (필수) | 결제 감지 → 조건 달성 → 보상 지급 코어 로직 | 핵심 기능 |
| P0 (필수) | EventManagerData 연동 (기간/스코프 관리) | 이벤트 운영 기반 |
| P0 (필수) | CSV 보상 테이블 | 데이터 드리븐 운영 |
| P1 (중요) | 이벤트 팝업 UI + 로비 버튼 | 유저 접근성 |
| P1 (중요) | 미수령 보상 우편 발송 | 보상 유실 방지 |
| P1 (중요) | AlarmUI 레드닷 알림 | 유저 인지 |
| P2 (선택) | 이벤트 종료 임박 알림 | 결제 유도 강화 |

## 4. 개요

### 기본 구성
- **이벤트 등록**: EventManagerData에 `EventInfoType.ConditionReward` 타입으로 이벤트 등록
- **조건 정의**: `ConditionEventRewardData` CSV 테이블에서 조건 타입/값과 보상을 정의
- **기간 관리**: `EventScopeType.Personal` + `start_day_offset` / `end_day_offset`로 계정 생성 기준 기간 운영
- **보상 지급**: 서버 판정 후 `CommonRewardUI`로 즉시 표시, 미수령 시 우편 발송

### 핵심 규칙
- 이벤트 기간 내 **조건 1회 달성** 시 보상 수령 가능 (단일 보상)
- 첫 결제 인스턴스: `ConditionType.First_Purchase` — BuyType.InApp인 **어떤 상품이든** 최초 1회 결제 시 달성
- 조건 달성 후 보상 미수령 상태에서 이벤트 종료 시 → 우편으로 자동 발송
- 이벤트 기간 전/후에는 로비 버튼 비노출

### 진입 조건 / 해금 조건
- **이벤트 시작**: 계정 생성 즉시 (start_day_offset = 0)
- **이벤트 종료**: EventManagerData의 `end_day_offset`에 설정된 일수 경과 시 (예: 7일)
- **해금 조건**: 없음 (계정 생성 시 자동 활성화). 필요 시 EventManagerData의 `unlock_condition_type`으로 추가 가능

### 보상 구조 요약
- 보상 슬롯 3개 (RewardType + reward_id + reward_amount) × 3
- CSV에서 자유 세팅 (골드, 재화, 유닛, 장비, 소모품 등)
- 예시: 재화 500 + SR 유닛 선택권 1 + 골드 50,000

## 5. 시스템 동작 흐름

### 전체 흐름 다이어그램
```
[계정 생성] → [이벤트 자동 시작 (Personal scope)]
     ↓
[로비 진입] → [이벤트 기간 확인] → [버튼 표시 + 알림]
     ↓
[유저 결제] → [서버 결제 완료] → [조건 달성 판정 (서버)]
     ↓
[클라이언트 통보] → [자동 보상 수령 (CommonRewardUI)]
     ↓
[이벤트 종료 시] → [미수령 보상 우편 발송 (서버)]
```

### 단계별 상세 흐름

#### Step 1. 이벤트 시작 (진입)
- 계정 생성 시 서버가 EventManagerData 기반으로 이벤트 인스턴스 생성
- `EventScopeType.Personal` → 계정 생성일 + `start_day_offset` ~ `end_day_offset` 기간 계산
- 클라이언트: `UserDataManager.UpdateEventData()` → `UserEventDatas`에 캐싱

#### Step 2. 로비 UI 표시
- 로비(`MenuUI`) 진입 시 `UserEventDatas.GetEventDataToEnable(eventIdx)` 호출
- 이벤트 활성 상태이면:
  - 독립 이벤트 버튼 표시 (기간 내만)
  - 보상 수령 가능 시 `AlarmUI` 레드닷 표시
  - 남은 기간 표시

#### Step 3. 조건 확인 (결제 감지)
- 유저가 **아무 InApp 상품** 결제 → `WASClient.ShopBuySend()` 호출
- 서버 응답에 이벤트 조건 달성 정보 포함
- 클라이언트: `MessageManager` 이벤트로 조건 달성 통보
  - 신규 메시지 코드: `eMESSAGE_CODE.ConditionRewardEventComplete`

#### Step 4. 보상 수령 (자동)
- 조건 달성 통보 수신 시 자동으로 보상 수령 API 호출
- 서버 응답: `ServerResultAddRewardUIData` 형태로 보상 데이터 전달
- `DefsReward.UpdateAddRewardUIData()` → `CommonRewardUI` 팝업으로 보상 표시
- 이벤트 상태를 "완료"로 갱신

#### Step 5. 미수령 보상 처리 (우편)
- 이벤트 종료 시점에 서버 배치 처리:
  - 조건 달성했으나 보상 미수령 유저 → 우편(`Mail`)으로 보상 자동 발송
  - 조건 미달성 유저 → 아무 처리 없음 (이벤트 종료)

### 예외 처리 / 엣지 케이스
| 상황 | 처리 방식 |
|------|----------|
| 결제 후 앱 종료 (보상 팝업 미표시) | 다음 로비 진입 시 보상 수령 가능 상태 확인 → 자동 수령 재시도 |
| 이벤트 기간 만료 + 보상 미수령 | 서버 배치에서 우편 자동 발송 |
| 이벤트 기간 만료 + 조건 미달성 | 이벤트 종료, 보상 없음 |
| 결제 환불 | 서버에서 환불 감지 시 조건 달성 취소 (보상 이미 수령 시 별도 정책 필요 — 프로그래머 논의) |
| 동일 이벤트 보상 중복 요청 | 서버에서 1회 지급 이력 확인 후 거부 |
| 이벤트 중복 운영 (같은 타입 여러 인스턴스) | EventManagerData idx로 구분, 각각 독립 운영 |

## 7. 밸런스 계획

### 설계 기준
- 첫 결제 보상은 "결제해서 좋았다"는 경험을 줄 수 있을 정도로 가치 있어야 함
- 단, 게임 밸런스를 크게 깨지 않는 범위 내에서 설정
- 구체적 수치는 CSV에서 기획자가 조정

### 주요 수치 테이블
| 항목 | 기준값 | 최솟값 | 최댓값 | 비고 |
|------|--------|--------|--------|------|
| 이벤트 기간 | 7일 | 3일 | 30일 | EventManagerData.end_day_offset |
| 보상 슬롯 수 | 3개 | 1개 | 3개 | ConditionEventRewardData |
| 우편 보상 만료 | 30일 | 7일 | 90일 | ConstantDatas |

### 성장 곡선 / 단계별 수치
| 단계 | 조건 | 수치 | 비고 |
|------|------|------|------|
| 단일 | 첫 InApp 결제 (금액 무관) | 보상 1세트 | CSV에서 보상 내용 세팅 |

### 검증 기준
- 무과금 기준 달성 목표: 해당 없음 (과금 조건)
- 소과금 기준 달성 목표: 최소 금액 상품 1회 구매로 달성 가능
- 고과금 기준 달성 목표: 동일 (첫 결제 1회면 충분)

## 8. UI / UX

### 화면 목록
| 화면명 | 진입 경로 | 역할 |
|--------|----------|------|
| ConditionRewardEventPopUpUI | 로비 → 이벤트 버튼 | 이벤트 정보 + 보상 미리보기 + 달성 상태 표시 |
| CommonRewardUI (기존) | 보상 수령 시 자동 | 획득 보상 표시 팝업 |

### 화면별 상세

#### 로비 이벤트 버튼
- **위치:** 로비 화면 우측 또는 상단 독립 버튼
- **표시 조건:** 이벤트 기간 내 + 이벤트 활성 상태
- **주요 요소:**
  - 이벤트 아이콘 (icon_path/icon_name)
  - 남은 기간 텍스트 ("D-3")
  - AlarmUI 레드닷 (보상 수령 가능 시)
- **비표시 조건:** 이벤트 기간 외 또는 보상 수령 완료

#### ConditionRewardEventPopUpUI (이벤트 팝업)
- **레이아웃:**
  - 상단: 이벤트 제목 + 남은 기간
  - 중앙: 보상 아이템 미리보기 (아이콘 + 수량) × 3슬롯
  - 하단: 상태 표시 + 액션 버튼
- **주요 요소:**
  - 보상 슬롯 (RewardData 기반 아이콘/수량 표시)
  - 달성 상태 표시:
    - 미달성: "첫 결제 시 보상 수령!" + [상점 이동] 버튼
    - 달성 + 미수령: [보상 받기] 버튼 (활성)
    - 수령 완료: "보상 수령 완료" 텍스트 (비활성)
  - 남은 기간 카운트다운
- **인터랙션:**
  - [상점 이동] → ShopUI 오픈
  - [보상 받기] → 서버 API → CommonRewardUI 팝업
  - [닫기] → 팝업 닫기
- **참고 이미지/레퍼런스:** 기존 BeginnerEvent 팝업 레이아웃 참조

### UX 주의사항
- 결제 직후 자동 보상 팝업이 뜨므로, 이벤트 팝업을 별도로 열 필요 없이 `CommonRewardUI`로 바로 표시
- 이벤트 기간 종료 D-1 시점에 푸시 알림 발송 (미달성 유저 대상)
- 보상 수령 완료 후에도 이벤트 기간 내에는 버튼 유지 (완료 상태 표시), 또는 즉시 숨김 — 운영 판단

## 부록: 재활용 가이드

### 새로운 조건 달성 이벤트 추가 시
1. **EventManagerData** CSV에 신규 row 추가 (event_type = `ConditionReward`, 신규 event_id)
2. **ConditionEventRewardData** CSV에 신규 row 추가 (condition_type 변경, 보상 세팅)
3. 코드 변경 없이 운영 가능

### 재활용 예시
| 이벤트명 | condition_type | condition_value1 | 비고 |
|---------|---------------|-----------------|------|
| 첫 결제 이벤트 | First_Purchase | 0 | 금액 무관 첫 InApp 결제 |
| 첫 던전 클리어 | Dungeon_Clear | 1 | 던전 ID 1 클리어 |
| 스테이지 100 클리어 | Stage_Clear | 100 | 스테이지 100 달성 |
| 유닛 10체 수집 | Unit_Count | 10 | 보유 유닛 10체 이상 |

> ConditionType에 해당 타입이 이미 존재하면 코드 추가 없이 CSV만으로 운영 가능.
> 신규 조건 타입이 필요하면 ConditionType enum 추가 + 서버 판정 로직 추가 필요.
