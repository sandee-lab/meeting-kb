## 2. 달성 목표

### 플레이어 경험 목표
- 튜토리얼 직후 "이 재화들이 있으면 훨씬 수월해진다"는 필요성을 직접 체감하게 한다
- 소액부터 시작해 "이 정도면 살 만하다"는 첫 결제 심리적 허들을 낮춘다
- 3일간 보상이 이어지는 구조로 초반 접속 루틴을 형성한다

### BM 목표
- 신규 유저의 **첫 과금 전환율** 극대화 (1단계 1,500원 허들 최소화)
- 3단계 상품 동시 구매를 통한 **초기 ARPU 극대화**
- 고가 상품(3단계 전설 유닛)을 통한 **고과금 유저 조기 확보**

### 운영/개발 목표
- 구매 기간 제한 없이 지속 노출 → 리텐션 구간(D+7~30)에서도 지속적인 전환 기회 확보
- 팝업 내 직접 수령 방식으로 유저 행동 유도 + 서버 구현 복잡도 최소화

## 3. 수행 방향

### 핵심 설계 원칙
1. **낮은 첫 과금 허들** — 1단계를 1,500원으로 설정하여 "일단 한 번" 결제를 유도
2. **상품 간 독립성** — 3개 상품은 각각 독립 구매 가능. 하나를 사도 나머지는 계속 노출
3. **가치 선지급** — 구매 즉시 100% 가치를 전달하여 결제 만족도 확보. 2~3일차 보상은 추가 보너스로 설계
4. **접속 유도 설계** — 매일 최초 접속 시 팝업이 뜨고, [수령] 버튼을 눌러 보상을 직접 수령하는 구조. 수령 후 버튼은 [내일 수령 가능]으로 교체되어 다음 접속 동기를 부여 (콩콩 던전 스타일)
5. **이월 보장** — 누적 출석 기준 지급으로 소멸 불안감 제거 → 유저 신뢰도 확보

### 제약 사항 / 금지 사항
- 구매 후 환불 시 이미 지급된 보상 처리 정책 사전 정의 필요
- 동일 계정 중복 구매 불가 (각 단계별 1회 한정)

### 우선순위
| 우선순위 | 항목 | 이유 |
|---------|------|------|
| P0 (필수) | 구매 즉시 1차 보상 우편 지급 | 핵심 가치 전달 |
| P0 (필수) | 튜토리얼 종료 후 팝업 자동 노출 | 진입점 |
| P0 (필수) | 메인 HUD 좌측 재진입 버튼 | 재방문 경로 |
| P1 (중요) | 누적 출석 기반 2·3일차 보상 팝업 수령 | 리텐션 유도 |
| P1 (중요) | 보상 수령 시 공용 보상 연출 창 표시 | 수령 체감 |
| P2 (선택) | 마지막 보상 수령 시 탭 소멸 연출 | 완료 피드백 |

## 4. 개요

### 기본 구성

| 단계 | 가격 | 주요 보상 재화 | 비고 |
|------|------|--------------|------|
| 1단계 | 1,500원 | 유닛 소환권 + 소량 다이아 | 첫 과금 허들 전환 목적 |
| 2단계 | 7,500원 | 장비 소환권 + 특정 등급 장비 | 중간 가치 제공 |
| 3단계 | 21,000원 | 전설 유닛 (고정 ID 지정) + 전설 유닛 조각(추가 성장) | 고과금 유저 조기 확보 |

### 핵심 규칙
- 3개 상품은 **독립 구매** 가능 (순서 무관, 중복 불가)
- 각 상품 구매 시 **3회 보상** 지급 (즉시 / 누적 출석 2일차 / 누적 출석 3일차)
- 보상은 **팝업 내 [수령] 버튼**을 통해 직접 수령 (공용 보상 연출 창 표시)
- 수령 후 버튼은 **[내일 수령 가능]** 텍스트로 교체 (다음 누적 출석일까지 비활성)
- 구매 즉시 지급 보상 = **구매 가치의 100%**. 2·3일차 보상은 추가 보너스
- 마지막 보상(누적 3일차) 수령 시 해당 상품 탭 소멸. **모든 상품 탭 소멸 시** 팝업 자동 닫기 + HUD 버튼 제거

### 진입 조건 / 해금 조건
- **최초 노출:** 컨텐츠 언락 시스템을 활용하여 원하는 타이밍에 팝업 노출을 제어 (하드코딩이 아닌 데이터 기반 타이밍 제어)
- **수동 재진입:** 메인 HUD 좌측 버튼 → 미구매 상품이 남아있는 동안 항상 노출
- **팝업 우선순위:** 아래 기존 팝업 노출 우선순위 큐 기준, **초보자 패키지는 시즌 패스 시작 연출 바로 앞**에 위치

**현재 팝업 우선순위 큐 (위가 높은 순위)**

| 순서 | 팝업 |
|------|------|
| 1 | 레벨 업 |
| 2 | 콘텐츠 언락 |
| 3 | 튜토리얼 |
| 4 | 선착순 랭킹 안내 |
| 5 | 최초 클리어 상품 구매 팝업 |
| 6 | 엔딩 챕터 클리어 축하 팝업 |
| 7 | (디폴트 값) |
| **8** | **초보자 패키지** |
| 9 | 시즌 패스 시작 연출 |
| 10 | 멤버십 만료 |

### 보상 지급 기준 — 누적 출석 정의

| 보상 차수 | 지급 조건 | 지급 방식 | 수령 후 상태 |
|----------|----------|----------|-------------|
| 1차 | 구매 즉시 | 팝업 내 [수령] 버튼 → 공용 보상 연출 창 | [내일 수령 가능] 표시 |
| 2차 | 구매 후 누적 출석 2일차 접속 | 팝업 내 [수령] 버튼 → 공용 보상 연출 창 | [내일 수령 가능] 표시 |
| 3차 | 구매 후 누적 출석 3일차 접속 | 팝업 내 [수령] 버튼 → 공용 보상 연출 창 | **해당 상품 탭 소멸** |

> ※ 누적 출석 기준이므로, 하루를 건너뛰어도 보상은 소멸되지 않고 이월됩니다.
> ※ 마지막 보상 수령 시 해당 탭이 소멸되며, 남은 상품이 없으면 팝업 자체가 닫힙니다.

## 5. 시스템 동작 흐름

### 전체 흐름 다이어그램

```
[컨텐츠 언락 조건 충족]
      ↓
[초보자 패키지 팝업 표시]
      ↓
[유저 행동 분기]
  ├─ [팝업 닫기] → [메인 HUD 좌측 버튼 노출] → [버튼 터치 시 팝업 재표시]
  └─ [상품 선택 후 [구매] 버튼 터치]
            ↓
      [구매 확인 팝업 (3일 합산 보상 미리보기 + 청약 철회 문구)]
            ↓
      [결제 진행 / 취소]
            ↓
      [결제 성공 확인]
            ↓
      [1차 보상 [수령] 버튼 활성화] → [수령] → [공용 보상 연출 창]
            ↓
      [버튼 → [내일 수령 가능] 교체]
            ↓
      [누적 출석 2일차 접속] → [2차 보상 [수령] 버튼 활성화] → [수령] → [공용 보상 연출 창]
            ↓
      [누적 출석 3일차 접속] → [3차 보상 [수령] 버튼 활성화] → [수령] → [공용 보상 연출 창]
            ↓
      [해당 상품 탭 소멸]
            ↓
      [남은 상품 탭 없음] → [팝업 자동 닫기 + HUD 버튼 제거]
```

### 단계별 상세 흐름

#### Step 1. 노출 타이밍 감지
- 컨텐츠 언락 시스템의 조건 충족 여부 확인 (노출 타이밍 데이터 기반 제어)
- 초보자 패키지 팝업 UI 호출
- 미구매 상품이 1개 이상 남아있는 경우에만 노출

#### Step 2. 팝업 표시 및 상품 선택
- 3개 상품을 나란히 표시
- 이미 구매 완료(3차 보상까지 수령)된 상품은 **해당 탭 자체를 제거** (재구매 불가)
- 각 상품의 가격, 주요 보상 구성, 3일차 보상 예시 표시

#### Step 3. 구매 확인 팝업 (뎁스 +1)
- [구매] 버튼 터치 시, 초보자 패키지 팝업 위에 **구매 확인 팝업**을 오버레이 표시
- **3일간 합산 보상 미리보기**: 해당 상품의 1차·2차·3차 보상을 모두 합산하여 "3일간 접속하면 이 보상을 모두 받습니다" 안내
- **청약 철회 문구**: 팝업 하단에 청약 철회 관련 법적 고지 문구 표시
- [결제하기] 버튼 → OS 결제 플로우 진행
- [취소] 버튼 → 팝업 닫고 상품 선택 화면으로 복귀

#### Step 4. 결제 처리
- OS 네이티브 결제 플로우 실행 (iOS / Android)
- 결제 성공 → WAS 서버 검증 → 구매 완료 처리
- 결제 실패 → 안내 팝업 후 복귀

#### Step 5. 1차 보상 수령
- 결제 검증 완료 즉시 해당 상품 탭의 [수령] 버튼 활성화
- [수령] 터치 → **공용 보상 연출 창** 표시 → 보상 지급
- 수령 후 버튼은 **[내일 수령 가능]** 텍스트로 교체

#### Step 6. 누적 출석 기반 2·3차 보상 수령
- 서버에서 유저의 누적 출석일 추적
- 구매 시점 기준 누적 출석 2일차 접속 → 팝업 표시 → 해당 상품 탭의 [수령] 버튼 활성화
- 구매 시점 기준 누적 출석 3일차 접속 → 팝업 표시 → 해당 상품 탭의 [수령] 버튼 활성화
- 접속을 건너뛰어도 조건 충족 시점에 수령 가능 (이월 적용)
- 각 수령 시 **공용 보상 연출 창** 표시

#### Step 7. 완료 처리
- 3차 보상 수령 완료 → **해당 상품 탭 소멸**
- 남은 상품 탭이 없으면 → **팝업 자동 닫기 + HUD 버튼 제거**

### 예외 처리 / 엣지 케이스

| 상황 | 처리 방식 |
|------|----------|
| 결제 성공 후 네트워크 단절 | 서버 영수증 검증 후 재접속 시 해당 상품 탭에 [수령] 버튼 활성화 |
| 수령 버튼 터치 후 네트워크 단절 | 서버 보상 지급 확인 후 재접속 시 상태 동기화 (이미 지급됐으면 다음 단계로 이동) |
| 동일 상품 중복 결제 시도 | 구매 완료 상태로 처리, 결제 차단 |
| 1·2차 보상 수령 후 환불 | 기존 패키지 상품 환불 처리 구조 참조하여 동일 정책 적용. 분할 보상 이미 지급분 회수 범위 정의 필요 → `검토 필요` |
| 튜토리얼 스킵 (향후 도입 시) | 스킵 완료 시점을 튜토리얼 종료로 동일 처리 |
| 3단계 전설 유닛 이미 보유 시 | 중복 유닛 → 조각 자동 전환 기능 적용 여부 구현 확인 필요 |
| 매일 최초 접속 시 팝업이 뜨지 않는 경우 | HUD 버튼을 통해 수동 진입 가능. 수령 가능 상태면 [수령] 버튼 활성화 |
| 구매 후 보상을 수령하지 않고 여러 날 접속한 경우 | 누적 접속일 기준으로 수령 가능한 회차까지 모두 열어둠. [수령] 터치 시 열린 보상을 **일괄 지급** (공용 보상 연출 창에 합산 표시). 예: 3일 접속했으나 한 번도 수령 안 했으면 → 1차·2차·3차 보상 모두 열림 → [수령] 한 번으로 전부 지급 → 탭 소멸 |

### 확인 필요 사항 (프로그래머 논의)

| 항목 | 내용 | 상태 |
|------|------|------|
| 서버 API 엔드포인트 | 신규 API 필요 여부 vs 기존 `game/shop_buy` 플로우 재사용 | `미확정` |
| 보상 수령 트리거 | 클라이언트 접속 시 서버에 누적 출석일 요청 → 수령 가능 여부 판단 → [수령] 버튼 활성화 | `미확정` |
| 전설 유닛 중복 보유 처리 | 기존 중복 유닛 → 조각 변환 로직이 우편 지급에도 동일 적용되는지 | `구현 확인 필요` |
| 환불 정책 | 기존 패키지 상품(`ShopPackageData`) 환불 처리 구조를 참조하여 동일 정책 적용 여부 확인. 분할 보상(2·3일차) 이미 지급된 경우의 회수 범위 정의 필요 | `검토 필요` |

## 6. 데이터시트 구조

> ※ `ShopBeginnerEventPackageData`는 초보자 미션 일자별 언락 패키지용으로, 본 시스템과 용도가 다릅니다.

### 테이블 목록

| 테이블명 | 역할 | 구분 |
|---------|------|------|
| `ShopBeginnerAttendancePackageData` | 패키지 상품 정의 (가격, PID, 효율, 차수별 보상 pool 연결) | **신규 테이블** |
| `ShopPackagePoolData` | 차수별 보상 재화 상세 목록 | **기존 테이블 공용** |

### 테이블 상세

#### `ShopBeginnerAttendancePackageData` — 상품 정의 (신규)

> ※ 향후 UI에 상품 안내 문구를 표시하게 되면, 다국어 관련 컬럼(`local_desc` 등)이 추가될 수 있음. 현재는 기본 구조로 진행.

```csharp
// Assets/Scripts/CSV/ShopBeginnerAttendancePackageDatas.cs (신규 생성)
[Serializable]
public class ShopBeginnerAttendancePackageData : BaseShopData
{
    // BaseShopData 상속: product_id, order, local_name, icon_path, icon_name

    public int step;                // 패키지 단계 (1 / 2 / 3)

    public BuyType fee_type;        // 결제 방식
    public string aos_pid;          // Android 스토어 상품 ID
    public string ios_pid;          // iOS 스토어 상품 ID
    public int krw;                 // 원화 가격 (내부 로그용)
    public int purchase_limit;      // 구매 횟수 제한 (1)
    public int value;               // 효율 표시 값 (UI에서 "{value}%"로 출력. 예: 1000 → "1000%")

    public int pool_id_day_1;       // 1차 보상 pool (구매 즉시)
    public int pool_id_day_2;       // 2차 보상 pool (누적 출석 2일차)
    public int pool_id_day_3;       // 3차 보상 pool (누적 출석 3일차)
}

public class ShopBeginnerAttendancePackageDatas : BaseCSVDatas<ShopBeginnerAttendancePackageData>
{
}
```

**필드 설명**

| 필드 | 타입 | 역할 |
|------|------|------|
| `product_id` | int | 상품 고유 ID (BaseShopData 상속) |
| `order` | int | 팝업 내 표시 순서 (BaseShopData 상속) |
| `local_name` | string | 상품 표시명 다국어 키 (BaseShopData 상속) |
| `icon_path` / `icon_name` | string | 상품 아이콘 (BaseShopData 상속) |
| `step` | int | 패키지 단계. 1 = 1,500원 / 2 = 7,500원 / 3 = 21,000원 |
| `fee_type` | BuyType | 결제 방식 (IAP) |
| `aos_pid` / `ios_pid` | string | 스토어 인앱 상품 PID |
| `krw` | int | 원화 기준 가격 (내부 로그용, 실제 결제 금액은 스토어 PID 기준) |
| `purchase_limit` | int | 계정당 구매 제한 횟수 (1) |
| `value` | int | 효율 표시 값. UI에서 `"{value}%"`로 출력 (예: 1000 → "1000%") |
| `pool_id_day_1` | int | 1차 보상 (구매 즉시) → `ShopPackagePoolData.pool_id` |
| `pool_id_day_2` | int | 2차 보상 (누적 출석 2일차) → `ShopPackagePoolData.pool_id` |

## 7. 밸런스 계획

> ⚠️ 구체적인 보상 수치는 추후 검토 예정. 아래는 설계 방향 기준입니다.

### 설계 기준
- 1차 보상 = **구매 가치의 100%** (결제 즉시 체감 가치 확보)
- 2차 보상 = 추가 보너스 (~구매 가치의 30~50% 수준 검토)
- 3차 보상 = 추가 보너스 (~구매 가치의 30~50% 수준 검토)
- 총 지급 가치 = 구매 가격 대비 **180~200% 수준** 목표 (패키지 매력도 확보)

### 상품별 보상 방향

| 단계 | 가격 | 1차 보상 (즉시) | 2차 보상 (2일차) | 3차 보상 (3일차) |
|------|------|----------------|----------------|----------------|
| 1단계 | 1,500원 | 유닛 소환권 N장 + 다이아 N개 | 유닛 소환권 추가 | 다이아 추가 |
| 2단계 | 7,500원 | 장비 소환권 N장 + 특정 등급 장비 1개 | 장비 소환권 추가 | 장비 강화 재료 |
| 3단계 | 21,000원 | 전설 유닛 1마리 + 전설 유닛 조각 N개 | 전설 유닛 조각 추가 | 전설 유닛 조각 추가 |

> ※ 수치 미확정. 수치 확정 후 이 표를 업데이트할 것.

### 검증 기준
- 무과금 기준: 1단계만 구매해도 초반 덱 구성에 유의미한 도움이 되는 수준
- 소과금 기준: 1+2단계 구매 시 동일 무과금 대비 스테이지 진행 속도 체감 차이 발생
- 고과금 기준: 3단계 구매 시 전설 유닛 조기 확보로 채널 내 상위권 경쟁 진입 가능

  - 미구매 상품: [구매] 버튼
  - 구매 완료 & 수령 가능: [수령] 버튼
  - 구매 완료 & 수령 완료: [내일 수령 가능] 텍스트 (비활성)

  - 탭 전환 → 해당 상품 정보 표시

  - [수령] 버튼 터치 → **공용 보상 연출 창** 표시 → 보상 지급
  - 3차 보상 수령 시 → 해당 탭 소멸. 남은 탭 없으면 팝업 자동 닫기
- **참고:** 콩콩 던전 스타일 참고. UI 시안 문서 별도 정의
- **참고 이미지:** `Docs/01_System/images/초보자_패키지_UI_참고.png`

#### 구매 확인 팝업 (뎁스 +1)

  - [결제하기] 버튼
  - [취소] 버튼
  - **청약 철회 문구**: 팝업 하단에 법적 고지 문구 표시 (예: "구매 후 7일 이내, 미사용 시 청약 철회 가능")
- **인터랙션:**
  - [결제하기] → OS 네이티브 결제 플로우 → 결제 성공 시 팝업 닫기 + [수령] 버튼 활성화

#### HUD 버튼 알림 배지
- 수령 가능한 보상이 있는 경우, **HUD 초보자 패키지 버튼에 빨간 점 표시**
