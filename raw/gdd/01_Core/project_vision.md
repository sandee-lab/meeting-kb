# MiniCraft 프로젝트 비전

> **요약:** 게임 정체성, 기획 철학, 유저 여정 프레임워크. 모든 기획 작업의 방향성 기준점.
> **현재 상태:** 미적용

> **이 문서의 역할:** "우리 게임은 이런 게임이다" — 개별 기획서가 바뀌어도 흔들리지 않는 원칙과 방향. 모든 기획 스킬(`/기획서`, `/브레인스토밍`, `/시니어 리뷰어` 등)이 이 문서를 기준으로 정합성을 판단한다. 시스템별 스펙은 해당 기획서에, 이 문서에는 원칙만 남긴다.

---

## 우리가 만드는 게임

하이브리드 캐주얼 액션 디펜스 — 뱀파이어 서바이벌류 로그라이크 인게임 코어에 운빨존많겜·랜덤다이스의 머지/합성 전략이 결합된 새로운 형태의 게임. 플레이어는 웨이브 사이 전략 페이즈에서 유닛을 랜덤 생성하고 합성하며, 확률이 만드는 의외성과 도파민이 매 판의 핵심 경험이다.

**핵심 경험 루프:** 랜덤 생성 → 합성 판단 → 전설 유닛 도달 → 경쟁에서 검증
- 웨이브 중간 멈춤 → 전략 수정 → 다음 웨이브의 리듬이 "한 판"의 재미
- 방치형 보상 루프가 **매일 접속하는 이유**, 경쟁이 **성장하는 이유**
- 60챕터 졸업 구조를 중심으로 "내가 이길 수 있다"는 성취감을 자주 경험하도록 설계

**핵심 재미 4요소:**
1. 다수의 몬스터를 처치하는 액션의 쾌감 (뱀파이어 서바이벌류)
2. 유닛 랜덤 생성과 합성이 만드는 의외성의 도파민 (랜덤다이스·운빨존많겜류)
3. 머지를 통해 전설 유닛까지 확률적으로 도달하는 수집의 재미
4. 전술카드와 덱 조합으로 각기 다른 전략을 만드는 빌드의 재미

**최종 목표:** 각 덱마다 독특한 컨셉을 갖고, 다양한 전략으로 즐길 수 있는 게임

**레퍼런스:** survivor.io, 카피바라 go, 궁수의 전설, 랜덤다이스, 운빨존많겜 — 하비 스타일 게임의 컨텐츠 방향성을 따라가되 우리만의 감성을 만든다.

## 기획 철학

### 인게임 원칙

- **전설 유닛이 덱의 핵심이다.** 전설 유닛은 각 속성·컨셉 덱의 중심축이며, 나머지 등급의 유닛은 그 덱을 완성하는 시너지 조각이다. 전설 유닛 조합이 모든 성장의 최종 목표
- **챕터 진행이 새로운 조합을 열어준다.** 챕터가 진행됨에 따라 더 강력한 덱을 조합할 수 있어야 하며, 신규 유닛 추가가 곧 새로운 메타를 창출한다
- **전투력이 전략에 선행한다.** 모바일 게임 특성상, 다양한 전략을 시도하기 전에 충분한 전투력 확보가 필요한 구조로 설계한다. 전략의 깊이는 전투력 기반 위에서 열린다
- **성장 축은 분산시키되 겹치지 않게.** 각 축(유닛, 장비, 배틀슈트, 골드 성장, 유물 등)은 독립적인 역할을 갖고 서로 대체하지 않는다
- **경쟁은 아웃게임 성장의 검증 창구.** 덱 전투력이 결과를 결정하며, 성장에 의미를 부여하는 구조
- **기존 자산 재활용 우선.** 신규 모드·콘텐츠는 기존 보스, 맵, 오브젝트를 최대한 활용

### 아웃게임 원칙

- **레퍼런스를 충분히 분석하고 검증된 방식을 적용한다.** 아웃게임에서는 우리만의 스타일보다 업계(하비, 중국 게임들)에서 검증된 구조를 채택하는 것이 중요하다
- **덱 다양성이 과금 뎁스를 넓힌다.** 다양한 덱을 활용할 수 있는 콘텐츠를 설계해야 과금의 폭을 옆으로 확장할 수 있다. 인게임에서 저장한 덱을 아웃게임에서 활용하는 구조가 이를 뒷받침한다
- **단계적 해금, 다양한 콘텐츠.** 복잡한 시스템은 후반에 열리도록 하되, 가급적 다양한 콘텐츠를 제공한다
- **보상 이월을 보장한다.** 미수령 보상은 우편 자동 발송. 유저의 노력이 사라지지 않는다는 신뢰
- **서버 검증 필수.** 모든 보상·결제 판정은 서버 주도, 클라이언트는 표시·요청만
- **범용 시스템을 만들어 조건만 바꿔 재활용한다.** 한 번 만든 이벤트·패키지·패스 프레임워크를 CSV 설정만으로 다양한 인스턴스에 재사용

### BM 원칙

- **레퍼런스가 요구하는 수준을 따라간다.** 경쟁 타이틀 대비 빈약한 BM은 허용하지 않는다
- **캐주얼한 겉모습, 깊은 과금 뎁스.** 캐주얼 비주얼이기 때문에 중·소과금 유저를 빠르게 과금으로 안착시키는 것이 핵심이다. 첫 과금 허들을 최소화하고 PUR을 최대한 높이는 전략을 취한다
- **가벼운 경쟁이 과금의 다리가 된다.** 라이벌 랭킹, 주간 랭킹, 라바퀘스트처럼 북미 캐주얼 게임들이 쓰는 부담 없는 경쟁 요소로 중과금에서 고과금으로 자연스럽게 전환한다. 추후 길드전 등 커뮤니티 경쟁으로 핵과금 영역까지 안착시킨다
- **"지금 사면 딱 좋다"는 타이밍 만족감.** 유저 행동 기반 니즈 시점에 노출
- **패키지가 다이아 직접 구매보다 압도적으로 유리하도록.** 다이아 단독 졸업 과금 = 효율 대비 5~10배
- **확정 성장 보장.** 성장 패키지는 구매 즉시 다음 마일스톤 도달이 가능하도록 구성

## 유저 여정 프레임워크

| 단계 | 목적 | 역할 |
|------|------|------|
| 1. 신규 진입 | 첫 결제 허들 최소화 | 소액 패키지로 과금 경험 시작, 접속 루틴 형성 |
| 2. 첫 결제 전환 | 무과금→소과금 전환 | 행동 트리거 기반 핫딜·이벤트로 자연스러운 첫 결제 유도 |
| 3. 중기 성장 | 확정 성장 체감 | 마일스톤 기반 패키지로 "구매 = 즉시 성장" 경험 |
| 4. 뽑기 목표화 | 뽑기를 도박→목표 활동으로 전환 | 누적 뽑기에 연동되는 보상 트랙으로 "꽝이어도 손해 없음" |
| 5. 과금 습관 형성 | 소액 구매 습관화 | 기간 한정 충전 이벤트로 일일 과금 루프 → 이후 월정액으로 자연 전환 |
| 6. 장기 성장 | 영구 누적 성장 동기 | 수집형 시스템으로 매일 조금씩 강해지는 체감, 만렙 이후 상위 성장 축 확장 |
| 7. 성취 피드백 | 성장 마일스톤 인지 강화 | 특별 강화·해금 시 연출로 성취감 + 성장 패키지 노출 기회 |
| 8. 경쟁 검증 | 성장 결과를 순위로 검증 | 주간 랭킹·리그 경쟁으로 아웃게임 성장에 의미 부여 |

> 각 단계의 상세 설계(가격, 트리거 조건, 보상 구성)는 해당 기획서를 참조: `Docs/01_System/`, `Docs/02_Unit/`

## 설계 가드레일

### 가격 범위 원칙

| 구간 | 범위 | 역할 |
|------|------|------|
| 첫 과금 | 1,000원대 | 심리적 장벽 최소화, 첫 결제 전환 |
| 중간 과금 | 3,000~7,000원대 | 성장 패키지, 중간 가치 제공 |
| 고가 과금 | 10,000원 이상 | 전설 유닛, 프리미엄 패스, 고과금 유저 확보 |

- 다국가 정규화를 위한 결제 포인트(charge_point) 시스템 적용
- 상세 가격표와 상품 구성은 개별 기획서에서 관리

### 보상 구조 원칙

- **범용 구조:** RewardType + reward_id + reward_amount. pool_id로 1:N 보상
- **신규 보상 설계 시:** 기존 기획서의 가격대·보상 수준과 정합성 확인 필수
- **보상 가치 기준:** 개별 기획서에 명시된 단가(예: 유물 조각 = 3,000원)를 레퍼런스로 참조

## 전략적 확장 방향

우리 게임이 장기적으로 나아가는 테마. 개별 기획서의 완료 여부와 무관하게 유효한 방향성:

- **속성별 전설 유닛 시리즈 확장** — 각 속성마다 고유 시너지 축을 갖는 전설 유닛을 추가하여 조합 깊이 확대
- **개인 경쟁 → 길드/소셜 → 서버 통합 경쟁** — 소규모 채널 경쟁에서 시작하여 경쟁 규모를 단계적으로 확장
- **초월 성장 축으로 만렙 이후 목표 제공** — 기존 시스템의 상위 성장 경로를 열어 충성 유저의 장기 목표 유지
- **시즌 경쟁 + 이벤트 로테이션으로 장기 운영** — 1~2주 단위 짧은 사이클로 신선함 유지, 단일 콘텐츠 피로도 방지
- **멤버십/구독으로 과금 안정화** — 일회성 패키지에서 정기 구독 상품으로 과금 패턴을 안정적으로 전환

## 조직의 특성

- 기획은 AI(Claude)와 기획자(김나은) + 개발 리드(이영섭)가 협업하여 작성
- AI가 초안을 생성하고 팀이 검토하거나, 개발 리드가 직접 기획서를 작성하기도 함
- 모든 기획서는 초안 상태로 시작하여 팀 리뷰를 거침
- 수치는 확정값이 아닌 방향성 — "세부 수치는 테스트 후 조정" 원칙
- 범용 시스템 설계를 선호 — 한 번 만든 프레임워크를 조건만 바꿔 재활용
- CSV 드리븐 운영을 강하게 지향 — 기획자가 코드 변경 없이 조정 가능한 구조
